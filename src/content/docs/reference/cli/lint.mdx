Run linting rules against your contract specs and surface violations before they reach a snapshot or release.

## Usage

```sh
contractual lint [--contract <name>] [--format <fmt>]
```

## Description

`contractual lint` iterates over every contract registered in `contractual.yaml` and runs the appropriate linter for its type. Contractual ships with sensible built-in linter defaults for each supported spec format:

| Spec type | Default linter |
|---|---|
| OpenAPI | `spectral lint` |
| JSON Schema | Built-in schema validator |
| AsyncAPI | `asyncapi validate` |
| ODCS | Built-in ODCS validator |

You can override any of these with a custom command via the `lint.command` field in `contractual.yaml`. See [Custom Linters and Differs](/guides/custom-governance) for details.

Violations are reported with file path, rule name, severity, and line number. The command exits with code `1` if any error-level violations are found, so it integrates cleanly into CI pipelines.

## Options

| Flag | Description | Default |
|---|---|---|
| `--contract <name>` | Run lint only for the named contract. Repeat to target multiple contracts. | All contracts |
| `--format <fmt>` | Output format. One of `text`, `json`. | `text` |

## Exit Codes

| Code | Meaning |
|---|---|
| `0` | All contracts passed linting with no error-level violations. |
| `1` | One or more error-level lint violations were found. |
| `2` | Configuration error — e.g. `contractual.yaml` not found or a contract name passed to `--contract` does not exist. |
| `3` | Tool error — the configured linter binary was not found or exited unexpectedly. |
