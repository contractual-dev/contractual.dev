Scan your project for spec files and generate the initial `contractual.yaml` configuration with baseline snapshots.

## Usage

```sh
contractual init [--dir <path>] [--format <fmt>]
```

## Description

`contractual init` walks your project directory looking for supported contract spec files — OpenAPI (`.yaml`, `.json`), JSON Schema, AsyncAPI, and Open Data Contract Standard (ODCS) files. For each spec it finds, it registers a contract entry in `contractual.yaml` and takes an initial snapshot.

Running `init` is always the first step in adopting Contractual in a project. It is safe to run in an existing repo: it will not overwrite a `contractual.yaml` that already exists.

After `init` completes:

- `contractual.yaml` is created (or left unchanged if it exists).
- `.contractual/` is created to hold snapshots and changesets.
- A baseline snapshot is written for every detected contract so that future `breaking` and `lint` runs have a reference point.

## Options

| Flag | Description | Default |
|---|---|---|
| `--dir <path>` | Root directory to scan for spec files. | `.` (current directory) |
| `--format <fmt>` | Output format for the command's own messages. One of `text`, `json`. | `text` |

## Exit Codes

| Code | Meaning |
|---|---|
| `0` | Init completed successfully. |
| `2` | Configuration error — e.g. `contractual.yaml` already exists, directory not found, or a spec file could not be parsed. |
